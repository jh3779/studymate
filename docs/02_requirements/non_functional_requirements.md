# 비기능 요구사항 정의서

## 1. 문서 목적

이 문서는 StudyMate 앱의 성능, 보안, 사용성, 유지보수성 등 기능 외 요구사항을 정의한다.

## 2. 핵심 내용

| 구분 | 요구사항 |
| --- | --- |
| 성능 | 일반 화면 전환은 지연 없이 동작해야 한다 |
| 안정성 | AI 요청 실패 시 앱이 종료되지 않아야 한다 |
| 보안 | AI API Key를 앱 내부에 하드코딩하지 않는다 |
| 데이터 보호 | 사용자별 데이터는 로그인 사용자 기준으로만 조회한다 |
| 사용성 | 학습 입력과 퀴즈 풀이 흐름은 3단계 이내로 이해 가능해야 한다 |
| 접근성 | 장애인, 고령자, 보조기술 사용자도 핵심 학습 흐름을 이용할 수 있어야 한다 |
| 유지보수성 | activities, models, services, utils, res/layout, res/drawable 구조로 분리한다 |

## 3. 상세 설명

### 성능

- 홈 화면과 학습 기록 목록은 Firestore 조회 후 로딩 상태를 표시한다.
- AI 요약과 퀴즈 생성은 외부 API 호출이므로 로딩 다이얼로그 또는 진행 표시를 제공한다.
- 퀴즈 화면은 이미 저장된 퀴즈 데이터를 사용하므로 문제 이동이 즉시 이루어져야 한다.

### 안정성

- 네트워크 오류, AI API 오류, JSON 파싱 오류를 처리한다.
- 오류 발생 시 사용자가 입력한 학습 내용은 보존한다.
- Firebase 인증 실패 시 원인을 알 수 있는 메시지를 표시한다.

### 보안

- Android 앱 내부에 AI API Key를 직접 작성하지 않는다.
- 실제 배포 시 Cloud Functions 또는 별도 백엔드를 통해 AI API를 호출한다.
- 수업 과제 MVP에서는 개발용 환경 변수 사용 방안을 문서화하고, 보안 리스크를 명시한다.

### 사용성

- 학습 내용 입력 화면은 제목, 과목명, 본문 입력 순서로 구성한다.
- 요약 결과 화면에서 바로 퀴즈 생성으로 이어질 수 있어야 한다.
- 결과 화면에서는 전체 문제 수, 맞힌 문제 수, 정답률을 한눈에 보여준다.

### 접근성

- 텍스트와 배경의 명도 대비를 충분히 확보한다.
- 오류, 성공, 선택, 정답, 오답 상태는 색상만으로 구분하지 않고 문구, 아이콘, 테두리를 함께 사용한다.
- 모든 입력 필드는 보이는 라벨을 제공하고 placeholder만으로 의미를 전달하지 않는다.
- 로그인과 회원가입은 비밀번호 관리자, 자동완성, 붙여넣기 사용을 막지 않는다.
- AI 요청 실패 시 사용자가 입력한 학습 내용은 보존한다.
- 정답률, 학습 통계, 오답 결과는 그래프 없이 텍스트만으로도 이해 가능해야 한다.
- 퀴즈 보기, 하단 탭, 주요 버튼은 충분한 터치 영역과 간격을 가진다.
- Android 구현 시 주요 조작 요소에 `contentDescription`과 명확한 텍스트 라벨을 제공하고 스크린리더 탐색 순서를 확인한다.
- 글자 크기 확대 설정에서도 학습 내용, 요약, 해설, 버튼 문구가 잘리지 않아야 한다.

### 유지보수성

- 화면 코드는 `Activity`, 화면 XML은 `res/layout`, 비즈니스 로직은 `services/` 성격의 Java 클래스로 분리한다.
- Firestore 문서 구조와 Java 모델 필드명을 최대한 일치시킨다.
- AI 프롬프트는 서비스 내부에서 상수 또는 별도 설정으로 관리한다.

## 4. 개발 시 참고사항

- 비기능 요구사항은 MVP에서도 누락되기 쉬우므로 오류 처리와 보안 주석을 우선 반영한다.
- API Key 보안은 수업 과제와 실제 배포 수준을 구분해 설명한다.
- Firestore 보안 규칙은 구현 단계에서 별도 작성해야 한다.
- 접근성 기준은 `docs/03_design/wireframe.md`의 접근성 고려사항을 함께 따른다.
